# Run Duration 64-bit Decision

## 结论

当前不升级 `run_duration` 到 64-bit。

## 原因

当前累计运行时长使用 `fan/run_s`，代码内存类型为 `uint32_t`，单位是秒。`uint32_t` 秒数上限约为 136 年，当前通过 Esp32BaseConfig `int32_t` API 持久化时的保守上限约为 68 年。

这已经远高于项目 NF-01 的 90 天连续运行目标，也高于当前可预期的 10 年使用周期。因此，当前不值得为了理论极限引入新的 NVS key、迁移流程和回滚复杂度。

## 保留现状

| key | 类型 | 说明 |
| --- | --- | --- |
| `run_s` | int | 累计运行秒数 |

代码保持：

- 内存字段：`uint32_t _run_duration`
- API 字段：`run_duration`
- 持久化 key：`fan/run_s`

## 后续触发条件

只有出现以下任一情况时，才重新评估 64-bit 迁移：

- 产品明确要求保存超过 50 年的生命周期累计秒数。
- Esp32BaseConfig 提供原生 `uint64_t` 配置 API。
- 运行时长统计从“维护观察值”升级为强业务账本资产。

## 注意

损坏的负数 `run_s` 仍应作为异常持久化输入单独处理：启动加载时需要先在有符号范围内校验，再转换为 `uint32_t`。
