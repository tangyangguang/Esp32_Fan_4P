# Run Duration 64-bit Migration Plan

## 背景

当前累计运行时长使用 `fan/run_s`，代码内存类型为 `uint32_t`，单位秒。该结构在长期连续运行目标下不够稳妥：90 天运行需求本身低于 `uint32_t` 上限，但未来累计统计会继续增长，且旧问题记录已明确要求在涉及持久化结构变化前先给出迁移方案。

## 目标

- 将累计运行时长升级为 64-bit，避免长期累计回绕。
- 保留现有设备上的 `fan/run_s` 数据，不清空用户统计。
- 升级失败时仍可读取旧值，不影响风扇控制主功能。

## 建议新结构

| key | 类型 | 说明 |
| --- | --- | --- |
| `run_s` | int | 旧累计秒数，只读迁移来源 |
| `run_hi` | int | 新累计秒数高 32 bit |
| `run_lo` | int | 新累计秒数低 32 bit |
| `run_v` | int | 运行时长持久化版本，`2` 表示已迁移 |

选择高低 32 bit 拆分，是因为当前 Esp32BaseConfig 已稳定使用 int API；本项目不应为单一业务字段引入自定义 NVS 访问绕开基础库。

## 升级流程

1. 启动时读取 `run_v`。
2. 如果 `run_v == 2`，读取 `run_hi/run_lo` 组合为 `uint64_t`。
3. 如果 `run_v != 2`，读取旧 `run_s`，转换为 `uint64_t`。
4. 首次保存时同时写入 `run_hi/run_lo/run_v=2`。
5. 暂时保留旧 `run_s`，只作为回滚和人工核对来源，不在迁移时删除。

## 回滚策略

- 旧固件仍可读取 `run_s`，但看不到新固件后的累计增量。
- 如必须回滚且要保留新累计值，可用 `run_hi/run_lo` 人工换算后写回 `run_s`；超过 `uint32_t` 上限时旧固件无法完整表示。

## 验收

- native 测试覆盖旧 `run_s` 迁移到 `run_hi/run_lo`。
- native 测试覆盖 `run_v=2` 直接加载新结构。
- `/api/status` 和 `/fan` 显示大于 `uint32_t` 的运行秒数时 JSON 不截断。
- `pio test -e native` 和 `pio run -e esp32dev` 通过。

## 实施状态

待用户确认后实施。当前版本没有静默修改持久化结构。
