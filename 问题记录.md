# 项目代码与文档评估问题记录

## 评估摘要
- 评估日期：2026-05-10
- 评估范围：`src/`、`test/`、`docs/`、`README.md`、`AGENTS.md`、`platformio.ini`、构建/测试/OTA 配置相关文件；当前仓库没有 `include/` 目录。
- 评估方式：静态代码阅读、文档交叉核对、Esp32Base 公开 API 边界抽查、`pio test -e native`、`pio run -e esp32dev`。
- 当前结论：复核成立且已修复的问题已从本文移除；当前没有仍待整改的 Critical/High/Medium 代码问题。

- 当前最大风险：剩余主要风险为实机验证和长期维护观察，包括 PWM 25 kHz/占空比、TACH RPM、堵转保护、软启动/停止、BOOT 人工长按和两键出厂重置。

## 低优先级 Low / 长期观察

### L-01 `run_s` 长期超过 int32 上限后会被持久化为负值
- 位置：`src/fan/FanController.cpp`、`docs/RUN_DURATION_DECISION.md`
- 现象：内存字段是 `uint32_t`，保存时受 Esp32BaseConfig int32 API 约束；超过 `INT32_MAX` 秒后，持久化值存在符号边界。
- 原因分析：项目已确认当前不升级 64-bit，并接受 int32 秒级保守上限约 68 年。当前 90 天和 10 年运行目标无实际风险。
- 影响范围：长期理论边界，不影响当前目标。
- 建议方向：保留 `RUN_DURATION_DECISION.md` 的触发条件；未来如果要求超过 50 年累计统计，再升级持久化结构。
- 是否需要实机验证：不需要。

### L-02 当前 Flash 使用率已到 OTA 分区的 81.8%，后续功能空间需持续观察
- 位置：`platformio.ini`、构建输出
- 现象：`pio run -e esp32dev` 显示 Flash 使用 1,071,541 / 1,310,720 bytes，约 81.8%。
- 原因分析：Full profile、Web、OTA、IRremoteESP8266 和业务页面共同占用较高。
- 影响范围：后续增加页面、协议库或诊断功能时可能逼近 OTA 分区上限。
- 是否属于基础库问题：否，当前不是 bug。
- 建议方向：后续每次较大功能后记录固件大小；若继续增长，评估分区表或裁剪依赖。
- 是否需要实机验证：不需要。

## 基础库 Esp32Base 反馈候选

### B-01 出厂重置需要可声明的基础库 namespace 清理范围
- 背景：本项目两键长按调用 `Esp32BaseConfig::clearLibraryNamespaces()` 完整出厂重置。
- 当前项目遇到的问题：项目文档只能依据当前 Esp32Base 实现描述 `eb_wifi`、`eb_sys`、`eb_log`、`eb_web`；未来基础库新增 namespace 后，本项目文档可能再次漂移。
- 为什么不应在本项目绕开：业务项目不应硬编码 `eb_*` namespace 列表，也不应直接操作基础库内部持久化结构。
- 建议反馈给 Esp32Base 的简洁提示词：为 Esp32Base 提供可枚举/可选择的出厂重置 API，明确 WiFi、Web Auth、系统、日志等 namespace 的清理范围，并提供 dry-run 或文档化清单，方便业务项目保护用户持久化资产。

### B-02 Config 审计 pre-begin 语义需要文档化
- 背景：本项目要求 `Esp32BaseConfig::enableConfigReadAudit(true)` 覆盖 `Esp32Base::begin()` 内部配置读取。
- 当前项目遇到的问题：代码已在 `Esp32Base::begin()` 前开启审计，但 Esp32Base 文档若不明确推荐时机，后续业务项目仍容易放到 begin 后。
- 为什么不应在本项目绕开：业务项目不应拆解 Esp32Base 启动流程或直接介入基础库内部读取。
- 建议反馈给 Esp32Base 的简洁提示词：请在 Esp32Base 文档中明确 `enableConfigAudit()` / `enableConfigReadAudit()` 可在 `Esp32Base::begin()` 前调用，并说明若需要覆盖基础库初始化读取，必须在 begin 前开启；补充一个最小示例。
