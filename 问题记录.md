## 一、严重（Critical）

### C-01 `setup()` 失败后 `loop()` 仍照常运行未初始化对象

- 位置：[src/main.cpp:98-118](src/main.cpp:98)
- 现象：`Esp32Base::begin()` 失败时 `setup()` 仅 `Serial.printf` 后 `return`，没有死循环或 `ESP.restart()`。Arduino runtime 之后照常进入 `loop()`，每次调用 `Esp32Base::handle()`、`fanController.tick()`、`handleBootButton()`。但 `fanController.begin()` 从未被调用，`FanDriver / ButtonDriver / LedIndicator / IRReceiverDriver` 都没有 `begin()`，更没有 `pinMode`/`attachInterrupt`/`ledcAttach`。
- 影响：未初始化外设状态下持续 tick，可能写无效 PWM、读未配置的 GPIO、调用未启动的 Web 模块；状态机以默认值 `_state=SYS_INIT` 进入 `_handleInit()`，理论上可恢复但 `Esp32Base::handle()` 行为未知。
- 建议：失败路径上明确 `while (true) { delay(...); }` 或 `ESP.restart()`，避免半初始化运行。

### C-02 `_handleError` 恢复窗口可被持续 `setSpeed` 无限重置

- 位置：[src/fan/FanController.cpp:211-222](src/fan/FanController.cpp:211)、[src/fan/FanController.cpp:429-444](src/fan/FanController.cpp:429)
- 现象：处于 `SYS_ERROR` 时，`setSpeed(>0)` 每次都会无条件 `_recovery_attempting = true; _recovery_start_tick = millis();`。如果用户（或前端轮询脚本）连续/快速调用 `setSpeed`，恢复窗口的 1500 ms 永远走不完，`_handleError` 中"RPM > 0 即恢复"的判定永远不触发，系统状态卡在 `SYS_ERROR`。
- 影响：实际风扇可能已正常旋转（`_fan.setSpeed` 在 `resetBlock` 后已经下发 PWM），但 UI 始终显示 Error/Blocked，干扰用户判断与日志分析。
- 建议：仅在 `_recovery_attempting == false` 或上一次窗口已结束时重置 `_recovery_start_tick`；或在 setSpeed 中检测 RPM 立即决定是否进入 RUNNING。

### C-03 错误恢复期间 system state 与 fan state 语义不一致

- 位置：[src/fan/FanController.cpp:211-222](src/fan/FanController.cpp:211)
- 现象：`SYS_ERROR` 中调用 `setSpeed(>0)` 会执行 `_fan.resetBlock(); _fan.setSpeed(speed);`，物理上风扇进入 SOFT_START / RUNNING，但 `_state` 没有改动，仍是 `SYS_ERROR`。前端 `/api/status` 返回 `state="Error"`，`/fan` 页面 `<b id=st>Error</b>`，用户看到错误但风扇却在转。
- 影响：可观测性混乱；自动化测试/脚本基于 state 判定时会误判。
- 建议：进入恢复窗口时引入 `SYS_RECOVERING` 子状态，或在 `_state` 上同步显式表达"恢复中"。

### C-04 `ESP32BASE_ENABLE_FILELOG` 守卫令配置审计可能被静默关闭

- 位置：[src/main.cpp:103-107](src/main.cpp:103)
- 现象：`#if ESP32BASE_ENABLE_FILELOG` 块同时包裹了 `Esp32BaseFileLog::enable(...)`、`Esp32BaseConfig::enableConfigAudit(true)`、`Esp32BaseConfig::enableConfigReadAudit(true)`。一旦该宏在某次 build/profile 调整中没有被定义为真，三件事会一起消失。
- 矛盾点：`AGENTS.md` §4 第 1 条要求 **`Esp32BaseConfig::enableConfigReadAudit(true)` 必须保持开启**——但这里完全依赖 Esp32Base 的宏定义，任何 profile 调整都可能违反"必须保持"的约束。
- 影响：配置审计、文件日志可能在不被察觉的情况下失效；事后定位现场问题缺少关键日志。
- 建议：将"审计开关"的开启与 FileLog 的开启解耦，至少把 `enableConfigReadAudit(true)` 放到 `#if` 之外。

## 二、高（High）

### H-01 BOOT 键消防（防误触）窗口仅 1 秒，触发出厂级动作

- 位置：[src/main.cpp:28](src/main.cps)、[src/main.cpp:57-85](src/main.cpp:57)
- 现象：`BOOT_CLEAR_WIFI_MS = 1000`，长按 GPIO0 仅 1 秒就触发 `Esp32BaseWiFi::clearCredentials()` + 重启。GPIO0 同时是 ESP32 boot strapping 引脚，DevKit 上极易在调试时被误碰。
- 影响：现场误触会丢失 WiFi 凭据，设备直接进入 AP 配网。AGENTS.md §10 要求"长按 BOOT 验证清 WiFi"是手工动作，但当前阈值偏低。
- 建议：阈值上调到 3–5 秒，并在按下到触发之间增加 LED 警示反馈（例如快闪倒计时）。

### H-02 BTN_BOTH_LONG 触发 `resetFactory` 同时清掉 WiFi/Auth

- 位置：[src/fan/FanController.cpp:266-275](src/fan/FanController.cpp:266)
- 现象：`resetFactory()` 调用 `Esp32BaseConfig::clearLibraryNamespaces()`，会一并清空 `eb_wifi.*`、`eb_web.auth_user/pass` 等基础库 NVS。等同于"恢复出厂"，文档（DOC-04 §第 7 节、AGENTS.md）只描述为"恢复出厂"，没明示连带清除范围。
- 影响：用户预期"恢复风扇默认配置"，实际包括 WiFi 凭据和 Web Auth；结合按键消抖期间的误判可能造成现场失联。
- 建议：在 README、DOC-01、DOC-04 明确"Both Long = 完全出厂复位（含 WiFi/Auth）"；或拆分两级：仅清 `fan` namespace vs 全清。

### H-03 风扇堵转检测时间默认 ≥ 软启动时间，恢复测试可能伪失败

- 位置：[src/fan/FanController.cpp:96](src/fan/FanController.cpp:96)、[src/fan/FanDriver.cpp:138-153](src/fan/FanDriver.cpp:138)、[src/fan/FanController.cpp:431-443](src/fan/FanController.cpp:431)
- 现象：默认 `block_detect_time=1500ms`，`soft_start_time=1000ms`，恢复窗口 1500ms（硬编码）。`FanDriver::tick()` 在 `FAN_STATE_RUNNING` 才检查堵转，软启动期间不计时；但软启动结束（约 1.0s）后才进入 RUNNING，剩 0.5s 即触发新堵转。`_handleError` 也恰好在 1.5s 后判 RPM——可能 RPM 还来不及更新（驱动 RPM 周期 500ms）。
- 影响：堵转/恢复时序与默认参数耦合脆弱，极易出现"恢复后立刻又被判堵转"，看上去像随机性 bug。
- 建议：恢复窗口至少要 ≥ `_soft_start_time + _block_detect_time + 500ms`；或恢复期间临时禁用堵转检测；硬编码 1500ms 改成派生值。

### H-04 `/api/status` JSON 写入固定 1024 字节缓冲，已逼近上限

- 位置：[src/web/FanWeb.cpp:365-422](src/web/FanWeb.cpp:365)
- 现象：单次 `snprintf` 写入约 24 个字段（含 IP、clock 字符串、`ir_last_code` 16 位 16 进制等）。粗略估算 800–950 字节，再加上 `state="Running"` 等小字段，已经接近 1024。任何字段（如 future hostname、profile 名称）增长都可能截断。
- 影响：截断后 JSON 缺尾，前端 `r.json()` 会抛异常，`/fan` 页面卡在轮询失败。
- 建议：扩大到 1536–2048，或改为 `sendChunk` 流式输出；或使用 `Esp32BaseWeb` 的 JSON helper（若有）。

### H-05 `run_duration` uint32_t 在 NF-01 90 天目标内会溢出

- 位置：[src/fan/FanController.h:102](src/fan/FanController.h:102)、[src/fan/FanController.cpp:387-394](src/fan/FanController.cpp:387)
- 现象：`_run_duration` 单位是秒、类型 `uint32_t`，最大 `4_294_967_295s ≈ 49.71 天`。需求 NF-01 要求 90 天连续运行，实测一旦风扇连续转满 49.71 天，累计时长就会回绕到 0，且 NVS 持久化也会被覆盖。
- 影响：S-01 累计运行时长在长期运行场景失真；现场维护人员看到清零会怀疑设备故障。
- 建议：升级为 `uint64_t`，或在显示时按"日 + 时 + 分"展示并落库小时数即可。

### H-06 `_handleInit` 与 `_sleep_entry_tick` 是死代码

- 位置：[src/fan/FanController.cpp:371](src/fan/FanController.cpp:371)、[src/fan/FanController.cpp:582](src/fan/FanController.cpp:582)
- 现象：`begin()` 已经把 `_state` 直接置为 `SYS_IDLE`，`tick()` 中 `case SYS_INIT` 永不进入；`_sleep_entry_tick` 只赋值不读取。
- 影响：维护负担、可读性下降，未来重构容易踩坑。
- 建议：删除 `_handleInit`（顺便从 `SystemState` 中考虑保留 `SYS_INIT` 与否），删除 `_sleep_entry_tick` 字段。

### H-07 `platformio.ini` 嵌入大量绝对路径，迁移困难

- 位置：[platformio.ini:14-18](platformio.ini:14)、[platformio.ini:48](platformio.ini:48)
- 现象：
  - `extra_scripts = post:/Users/tyg/dir/claude_dir/Esp32Base/scripts/esp32base_webota.py`
  - `lib_deps = symlink:///Users/tyg/dir/claude_dir/Esp32Base`
  - `board_build.partitions = /Users/tyg/dir/claude_dir/Esp32Base/partitions/esp32-4mb-ota-balanced.csv`
- 影响：换台机器或换 user 立即编译失败。CI、合作者、Esp32Base 路径调整都会卡住。
- 建议：把 `Esp32Base` 路径抽成 PlatformIO 变量（如 `ESP32BASE_DIR=${env.ESP32BASE_DIR}`），或改为相对路径（前提是仓库布局允许）。同时记录非 macOS 用户的接入方式。

## 三、中（Medium）

### M-01 `_handleSleep` 的"被唤醒"日志可能与状态不符

- 位置：[src/fan/FanController.cpp:409-422](src/fan/FanController.cpp:409)
- 现象：进入分支后先调用 `_processButtonEvents`/`_processIREvents`/`_processTimer`。这些回调可能已经把 `_state` 改成 `SYS_RUNNING`。随后只有 `if (_state == SYS_SLEEP)` 才会改回 `SYS_IDLE`，**但 `ESP32BASE_LOG_I("Woken from sleep, transitioning to IDLE")` 在 if 外面无条件输出**。
- 影响：日志显示"transition to IDLE"但其实跳到 RUNNING，事后排查 log 会困惑。
- 建议：把 log 移到 `if (_state == SYS_SLEEP)` 内部，或日志中带上当前 `_state`。

### M-02 状态卡片 HTML 在非堵转时输出空 class 属性

- 位置：[src/web/FanWeb.cpp:173-175](src/web/FanWeb.cpp:173)
- 现象：`snprintf(buf, sizeof(buf), "<div class='stat state'><span>State</span><b id=st class=%s>%s</b></div>", blocked ? "errtxt" : "", stateText);` —— 当 `blocked=false` 时输出 `<b id=st class= >Idle</b>`。HTML 校验会报错；浏览器虽然容错，但有些代理/扫描工具会告警。
- 影响：前端结构非标，影响可维护性与潜在 SSR 处理。
- 建议：在 blocked=false 时跳过 `class=` 整段；或始终给 class 一个有意义的占位（例如 `state-ok`）。

### M-03 `setSpeed`/`setTimer`/`stop` 每次都调用 `Esp32BaseWiFi::setPowerSave(false)`

- 位置：[src/fan/FanController.cpp:201](src/fan/FanController.cpp:201)、[src/fan/FanController.cpp:236](src/fan/FanController.cpp:236)、[src/fan/FanController.cpp:251](src/fan/FanController.cpp:251)
- 现象：每个外部命令都强制关掉 power save，无论是否处于 sleep。Web 频繁轮询 `/api/speed` 时会反复调用。
- 影响：在繁忙 API 场景下增加无谓系统调用；如果 Esp32BaseWiFi 实现非幂等还可能踩 race。
- 建议：只在 `_is_sleeping == true` 时调用一次；命令路径中只更新 `_last_operation_tick`，让 `_handleSleep` 唤醒时统一退出 power save。

### M-04 `_processSleep` 当 `_sleep_wait_time = 0` 时会立刻入睡

- 位置：[src/fan/FanController.cpp:565-590](src/fan/FanController.cpp:565)、[src/fan/FanController.cpp:343-347](src/fan/FanController.cpp:343)
- 现象：`setSleepWaitTime` 允许 0–3600；当为 0 时 `idle_time >= 0` 永远成立，进入 IDLE 后立即转 SLEEP。下次任意操作又被唤醒。
- 影响：逻辑可工作但与"等待秒数"语义违背；前端配置页 min=0 也未禁止该取值。
- 建议：把 `_sleep_wait_time` 最小值约束为 1（或在 0 时禁用 sleep 行为）。前端 min=1 同步。

### M-05 `LED_PWM_CHANNEL=1`、`PWM_CHANNEL=0` 与 Esp32Base 内部 LEDC 资源未做协调声明

- 位置：[src/fan/FanDriver.cpp:15](src/fan/FanDriver.cpp:15)、[src/fan/LedIndicator.cpp:19](src/fan/LedIndicator.cpp:19)
- 现象：FanDriver 显式占用 LEDC channel 0，LedIndicator 显式占用 channel 1。Arduino Core 3.x 走 `ledcAttach` 自动分配；Core 2.x 走显式编号。Esp32Base Full profile 是否在内部使用 LEDC（例如 buzzer/告警）当前不可知。
- 影响：如果 Esp32Base 后续也用 channel 0/1，会出现两路 PWM 相互覆盖；难以从日志诊断（仅表现为占空比异常）。
- 建议：在 DOC-02 / DOC-04 显式声明 LEDC 通道占用约束；在 LedIndicator/FanDriver 内增加可配置 channel 或 runtime 检查。本身可能也是 Esp32Base 提示词候选（基础库需提供"已用 channel 列表"）。

### M-06 `_runtime_save_interval_min` 持久化粒度可能与 `last_tim` 数据漂移共存

- 位置：[src/fan/FanController.cpp:604-617](src/fan/FanController.cpp:604)
- 现象：默认 1 分钟才落库一次，但 `last_tim` 也在这条路径上。断电时 `last_tim` 可能偏离实际剩余 0–60 秒；`run_duration` 同上。
- 影响：F-12 断电恢复在 timer 接近结束时可能多跑一段；S-01 累计运行损失最多一个 interval。
- 建议：把 `last_tim` 单独走更短间隔（例如每 10 秒）；或在 timer < 阈值时强制写入。

### M-07 文档列出的 `Esp32BaseSleep` / `Esp32BaseFs` / `Esp32BaseHealth` / `Esp32BaseBus` / `Esp32BaseMdns` / `Esp32BaseDns` / `Esp32BaseSystem` / `Esp32BaseWatchdog` 均未在源码中直接出现

- 位置：[docs/DOC-01_需求规格说明.md:88-101](docs/DOC-01_需求规格说明.md:88)、[docs/DOC-03_软件架构设计.md:67-83](docs/DOC-03_软件架构设计.md:67)
- 现象：DOC-01 §7 的"Esp32Base 验证清单"和 DOC-03 §4 的能力映射列出了 15 个模块；实际代码 `grep` 后只直接调用了 `Esp32Base`、`Esp32BaseConfig`、`Esp32BaseWeb`、`Esp32BaseWiFi`、`Esp32BaseNtp`、`Esp32BaseFileLog`、`Esp32BaseLog`。
- 影响：其他模块或许通过 `Esp32Base::begin()` 内部启用，但"业务侧实战验证"的成色不可见，验证清单与代码无法对账。
- 建议：在 DOC-01 §8 的需求追踪、DOC-04 §9 的当前状态中区分"显式 API 调用"与"由 begin() 隐式启用"，并补充对应实机验证证据（日志/接口）。

### M-08 README/AGENTS.md/DOC-04 的"当前状态"段已成长篇日志

- 位置：[README.md:9-95](README.md:9)、[AGENTS.md:31-34](AGENTS.md:31)、[docs/DOC-04_代码设计说明.md:156-198](docs/DOC-04_代码设计说明.md:156)
- 现象：多处反复描述同一批已验证项（OTA、AP 配网、health tick、ntp_sync_pending 降噪等），版本演进与"当前"区分不明。
- 影响：新 contributor 难以判断哪些是历史快照、哪些是当前实情；DOC-00 §2 表格与 README 状态段也存在重复。
- 建议：把"已验证记录"沉淀成 `docs/CHANGELOG.md` 或 `docs/VERIFY_LOG.md`，README/DOC-04 只保留指针；当前状态用版本号锚定。

### M-09 `kCaptureBufferSize` / `kCaptureTimeout` 放在头文件 file scope `static const`

- 位置：[src/fan/IRReceiverDriver.h:7-8](src/fan/IRReceiverDriver.h:7)
- 现象：两个 `static const` 在头文件 namespace 外，每个包含此头的 TU 都会得到独立副本。当前只有 IRReceiverDriver.cpp 用到，影响很小。
- 影响：风格瑕疵；如果未来在头里再加非 `static`/`inline` 的"常量"，更容易踩坑。
- 建议：移到 `IRReceiverDriver.cpp` 的匿名 namespace 内，或定义为 `class` 内 `static constexpr`。

## 四、低（Low）/ 风格类

### L-01 `parseUintArg` 的边界注释缺失

- 位置：[src/web/FanWeb.cpp:13-27](src/web/FanWeb.cpp:13)
- 现象：溢出判断 `parsed > (max_value - digit) / 10` 在数学上正确但难读。
- 建议：加一行简短注释或抽出常用的"checked add/multiply"工具。

### L-02 IR 学习失败 / 重复学习的 toast 未给出建议动作

- 位置：[src/web/FanWeb.cpp:255-267](src/web/FanWeb.cpp:255)
- 现象：当 `irDup>=0` 时仅提示"Already assigned to ..."，但不告知用户应当先 Clear 哪个键再学习。
- 建议：在 `setIr` 文案中追加"先 Clear 该键或换其他键"。

### L-03 `_loadConfig` 的 IR 解析失败时没有日志

- 位置：[src/fan/FanController.cpp:683-693](src/fan/FanController.cpp:683)
- 现象：若 NVS 中的 `ir_N` 字符串损坏，`parseIRCodeEntry` 失败，setKeyCode 不被调用，但**完全静默**。现场调试时无法判断是 NVS 损坏还是用户从未学习。
- 建议：解析失败处加 WARN 日志（带 key 名）；考虑写一次清理 entry。

### L-04 `LedIndicator` 在 `setOverride(LED_OFF)` 与"档位 0 关灯"语义重叠

- 位置：[src/fan/LedIndicator.cpp:56-78](src/fan/LedIndicator.cpp:56)
- 现象：`_base_mode` 同时承担"档位驱动"和"override"两个角色，互相覆盖时优先级隐式（最后一次 set 的胜出）。
- 影响：未来加新 override（例如 OTA 进行中）容易冲突。
- 建议：拆成 `_gear_mode` + `_override_mode` 两个字段，update 阶段合成最终输出。

### L-05 测试覆盖空白

- 位置：[test/test_fan_controller.cpp](test/test_fan_controller.cpp)
- 缺口：
  - `parseIRCodeEntry`（异常输入：空、缺冒号、超过 0xFFFFFFFFFFFFFFFF、protocol 越界）
  - `clearIRCode` 流程
  - `_handleError` 恢复成功 / 失败两条分支
  - `_handleSleep` 中按键唤醒
  - `handleBootButton` BOOT 长按时序
  - HTML 页面 chunk 顺序（仅当前 mock 的 `sendChunk` 是 no-op）
  - JSON 字段在大值（e.g. `run_duration > 1e9`）下不截断
- 建议：在 native 测试中扩展上述路径；HTML 输出可改 `sendChunk` 为收集到 buffer，再做关键串断言。

### L-06 `decode_results` 的 mock 与真库结构不同

- 位置：[test/IRrecv.h:18-22](test/IRrecv.h:18)
- 现象：mock 仅有 `decode_type` 与 `value`，真库还有 `repeat`、`bits`、`state[]` 等。代码中 `results.repeat` 只在 `#ifndef UNIT_TEST` 下访问，因此当前编译通过。但 mock 与真实结构进一步偏离时风险增大。
- 建议：在 mock 里补上 `bool repeat = false;`，删除源码侧的 `#ifndef UNIT_TEST` 包裹。

### L-07 `resetFactory` 始终返回 true

- 位置：[src/fan/FanController.cpp:266-275](src/fan/FanController.cpp:266)
- 现象：`clearNamespace` / `clearLibraryNamespaces` / `flushAll` 任一失败都只 ERROR 日志，不影响 `ESP.restart()`，函数最终 `return true`；调用方 `_processButtonEvents` 也不看返回值。
- 影响：函数签名带 `bool` 但语义死值；开发者误以为可用 ok 判断。
- 建议：要么改成 `void`，要么真正按 ok 决定是否重启（失败时不重启而是回滚到 IDLE 上等候人工干预）。

### L-08 `analogWrite` 在测试 mock 中代替 `ledcWrite`，但生产路径用的是 `ledcWrite`

- 位置：[src/fan/FanDriver.cpp:46-77](src/fan/FanDriver.cpp:46)、[test/test_fan_controller.cpp:24](test/test_fan_controller.cpp:24)
- 现象：`#ifdef UNIT_TEST` 下走 `analogWrite`，但 `g_pwmValue` 的语义/范围与 `ledcWrite` 在 8-bit 分辨率下虽然一致，硬件路径要走 LEDC channel + freq + resolution，三者只在硬件上才生效。
- 影响：测试断言 PWM 值可能误以为反映了硬件配置正确性，实际上 LEDC 频率/通道配置完全没被覆盖。
- 建议：在 mock 中追加 `ledcSetup`/`ledcAttach`/`ledcWrite` 的"调用记录"型替身，断言频率与 channel 设置。

### L-09 `Esp32BaseLog::INFO` 在 mock 中可见，但 `Esp32BaseLog` 类只暴露 enum，未定义构造或方法

- 位置：[test/Esp32Base.h:9-19](test/Esp32Base.h:9)
- 现象：仅 enum 占位，正常 `Esp32BaseFileLog::enable(..., Esp32BaseLog::INFO, 4)` 在 native 工程不参与构建（main.cpp 被 `build_src_filter` 排除）。这是设计；但当 mock 与真库后续分歧时，CI 不会暴露。
- 建议：把 `main.cpp` 的关键非平台代码（如配置审计开关）抽出来一起测；或在 mock 中追加 `Esp32BaseFileLog::enable` 的 spy。

### L-10 IR 学习成功 toast 用 `setTimeout(()=>location.reload(),2500)` 强刷页面

- 位置：[src/web/FanWeb.cpp:263](src/web/FanWeb.cpp:263)
- 现象：学习成功后 2.5 秒强制 `location.reload()`，刷掉用户输入与提示。
- 影响：体验断裂；如果用户连续学习多个键，会被刷新打断。
- 建议：直接通过 `/api/status` 拿最新 IR 列表，DOM 局部更新即可。

## 五、文档/约定一致性

### D-01 README "构建状态"清单与 DOC-00 §2 状态表内容大量重叠

### D-02 DOC-04 §1 配置表未列出 `led_ms`/`rt_save_m` 在 `/api/config` 与 `/api/status` 中的字段名

- 位置：[docs/DOC-04_代码设计说明.md:13-30](docs/DOC-04_代码设计说明.md:13)
- 现象：NVS 表中的 `led_ms` 对应 API 返回 `led_flash_ms`、`rt_save_m` 对应 `runtime_save_min`，文档未给出映射。
- 建议：补一列"API 字段名"，让前后端对齐有依据。

### D-03 README §"基础库使用原则"和 AGENTS.md §4 关于 enableConfigReadAudit 的强约束没有自动化保护

- 位置：[README.md:33-46](README.md:33)、[AGENTS.md:21-24](AGENTS.md:21)

- 建议：在 CI/native 测试中加一条断言：`enableConfigReadAudit` 必被调用（spy 计数）。

### D-04 DOC-02 GPIO 表对 GPIO2 strapping 风险说明不充分

- 位置：[docs/DOC-02_硬件设计说明.md:25-39](docs/DOC-02_硬件设计说明.md:25)

- 建议：在 GPIO2 备注里追加 strapping 行为提示，并给出"如果使用其他模组应改 GPIO"的兜底说明。

### D-05 DOC-01 §F-04 "响应 ≤ 200 ms"，但 ButtonDriver 默认消抖 50 ms，需求满足是默认隐含

- 位置：[docs/DOC-01_需求规格说明.md:24](docs/DOC-01_需求规格说明.md:24)、[src/fan/ButtonDriver.h:38](src/fan/ButtonDriver.h:38)
- 现象：消抖 50 ms + tick 周期取决于 loop()，实际响应应远小于 200 ms。但需求条款没有声明依据/测量方法。
- 建议：在 DOC-04 测试策略里加一条"按键消抖 + tick 时序的 200 ms 验证场景"。

### D-06 DOC-04 §5 "用户输入进入 JSON/HTML 前要转义"——目前业务侧未实现 HTML 转义

- 位置：[docs/DOC-04_代码设计说明.md:106](docs/DOC-04_代码设计说明.md:106)、[src/web/FanWeb.cpp:340-352](src/web/FanWeb.cpp:340)
- 现象：DOC-04 自己写"当前短字段先用数值输入，后续补全字符串转义"。`writeHtmlEscaped` 在 mock 头里有声明但业务代码完全没调用。当前 `irKeyName(i)`、`stateText` 都是常量字符串，安全；但一旦把用户可编辑字段（例如自定义 device name、IR 标签）引入页面，就有 XSS 风险。
- 建议：维持文档约束的同时，在引入第一处用户可编辑字段前先把 `Esp32BaseWeb::writeHtmlEscaped` 或等价 helper 接进来。

### D-07 ESP32BASE_PROMPTS.md 的"模板"未严格等同任何已有提示词样式

- 位置：[docs/ESP32BASE_PROMPTS.md:46-63](docs/ESP32BASE_PROMPTS.md:46)
- 现象：模板字段是中文章节标题，但 §"已完成基础库反馈记录"全部用 bullet 平铺，没遵循模板结构。新人写新提示词容易模板与示例不一致。
- 建议：保留模板，移除已完成记录中的"自由格式"重述，或加一句"以下为已完成项的简记，新提示词请按模板"。
